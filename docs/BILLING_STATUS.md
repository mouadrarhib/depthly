# Billing Status

Status snapshot of Depthly's billing/payments implementation. Not a tutorial — reflects the state as of 2026-07-16.

## Overview

Depthly uses **Lemon Squeezy** as the payment processor, not Stripe. Stripe does not support standalone merchant accounts in Morocco, so Lemon Squeezy (merchant-of-record model) was chosen instead.

`supabase/migrations/001_initial_schema.sql` and some older docs still use Stripe naming (`stripe_customer_id`, `stripe_subscription_id`, comments referencing "Stripe webhook"). This is expected and not a bug — the checkout/webhook Edge Functions built 2026-07-10 intentionally reuse these existing columns to store Lemon Squeezy IDs rather than renaming them; the rename is still deferred (see "Next steps" below).

- **Bug found and fixed 2026-07-14: `subscription_created` and `subscription_updated` were returning HTTP 500.** Confirmed in the Lemon Squeezy dashboard's webhook delivery log, and traced directly against live DB data: a new test subscription's row landed correctly in `subscriptions` (the audit-trail upsert has no relevant constraint), but the matching `profiles` row never updated. The webhook's profile update tried to write that already-claimed `customer_id` onto a second profile, threw, and the whole request 500'd — even though the event itself was legitimate. Fixed in `upsertSubscription()`'s profile update: on a `23505` violation, it now retries the same update without `stripe_customer_id`/`stripe_subscription_id`. Deployed 2026-07-14 (`lemonsqueezy-webhook` version 9).

- **Bug found and fixed 2026-07-15: Lifetime purchases never appeared in `BillingPage`'s billing history.** The billing history table reads the `subscriptions` table, but `order_created` (the event Lifetime purchases fire, since they're a one-time order rather than a subscription) only ever updated `profiles` — it had no `subscriptions` insert at all, unlike `subscription_created` which writes one via `upsertSubscription`. Fixed by having `order_created` also insert a `subscriptions` row for the Lifetime order: `plan: 'founding'`, `plan_interval: 'lifetime'`, `status: 'active'`, `current_period_start` from the order's `created_at`. Two schema-driven decisions worth noting: `stripe_subscription_id` is `NOT NULL` and `UNIQUE` on this table, and a one-time order has no real Lemon Squeezy subscription id, so the row uses `` `lifetime_order_${orderId}` `` there instead — prefixed so it can never collide with a real numeric subscription id (mirrors the convention `seed-demo-users.ts` already used for its own demo Lifetime rows). Likewise `current_period_end` is `NOT NULL`, so a purchase that never expires gets a sentinel far-future date (`2125-01-01`) rather than `null`; `BillingPage` tells a Lifetime row apart via `plan_interval`, not that column, and renders "Lifetime access" in place of a date range for it.
- **`order_refunded` handling added to `lemonsqueezy-webhook`: 2026-07-16.** Closes the gap flagged right after the July 16 live-mode test purchase was refunded — previously a real refund wouldn't have downgraded the customer's plan at all. On this event, the handler resolves the user the same way `subscription_expired` does: first via a matching `subscriptions` row (keyed by `` `lifetime_order_${orderId}` ``, the same key `order_created`'s Lifetime insert uses — this only matches Lifetime purchases, since a refunded Monthly/Yearly order is keyed by its subscription id instead, not available on this event), falling back to `custom_data.user_id`. It then sets `profiles.plan` to `'free'`, clears `plan_interval` and `is_founding_member`, and — if a Lifetime `subscriptions` row was matched — sets that row's `status` to the new `'refunded'` enum value (added by `supabase/migrations/005_add_refunded_subscription_status.sql`, **confirmed already applied to the live project** — queried directly via `pg_enum`) instead of deleting it, so billing history keeps an accurate record. Deliberately treats every refund the same way (Lifetime or subscription, full or partial) as a full downgrade for simplicity — noted in the code as needing refinement later if a partial refund on an otherwise-still-active subscription comes up in practice. Also noted as a known caveat: the `custom_data` fallback path could, in theory, incorrectly downgrade a founding member if the matched userId came from an unrelated old order refund rather than their own Lifetime purchase — not guarded against here, flagged for awareness rather than adding speculative branching.
- **Bug found and fixed 2026-07-16: `order_refunded` was written but never actually deployed, and even once deployed wouldn't have resolved a Monthly/Yearly refund's user.** A dashboard-simulated `order_refunded` test against a real Monthly/Yearly subscription returned HTTP 200 with no crash, but `profiles.plan` never changed. Investigation: downloading the actually-deployed function source (`supabase functions download`) and diffing it against the local file showed the live function had **zero occurrences of `order_refunded`** — the case above was written in the previous session but `supabase functions deploy` was never run afterward, so the live code fell through to the generic `default: break` case for this unrecognized event, silently doing nothing. Confirmed directly against the DB too: the target `subscriptions`/`profiles` rows were unchanged, still `active`/`pro`. Separately, even once deployed, the handler as written would likely have failed the same way for this exact scenario anyway: its only lookup path besides `custom_data` was the `lifetime_order_` key, which only ever matches one-time Lifetime purchases — a Monthly/Yearly subscription's row is keyed by its *subscription* id instead, and `order_refunded`'s payload doesn't carry that id inline (only `customer_id`, confirmed against real Lemon Squeezy Order attribute shapes fetched earlier). **Fixed with a third resolution path:** if neither the Lifetime-order match nor `custom_data` finds a user, the handler now looks up the most recently updated `subscriptions` row matching the order's `customer_id` (present directly in the flat order attributes) and uses that row's `user_id` for the profile downgrade only — deliberately not marking that row `'refunded'`, since it's a best-effort guess at "some subscription this customer has," not necessarily the exact one the order belongs to. **Deployed** (`lemonsqueezy-webhook` version 17, confirmed via `supabase functions list` and by re-downloading the deployed source and confirming the new code is present).

## What Exists Already in the Codebase

**Database (`supabase/migrations/001_initial_schema.sql`)** — modeled with Stripe naming:
- `plan_type` enum: `'free' | 'pro' | 'founding'`
- `plan_interval_type` enum: `'monthly' | 'annual' | 'lifetime'`
- `subscription_status_type` enum: `'active' | 'trialing' | 'past_due' | 'canceled' | 'unpaid'`, plus `'expired'` added by `supabase/migrations/004_add_expired_subscription_status.sql` and `'refunded'` added by `supabase/migrations/005_add_refunded_subscription_status.sql` — **both confirmed already applied to the live project**, queried directly via `pg_enum` (`select enumlabel from pg_enum e join pg_type t on t.oid = e.enumtypid where t.typname = 'subscription_status_type'` returns all 7 values including `expired` and `refunded`). This doc previously said `refunded` wasn't applied yet, which was stale.
- `profiles` billing columns: `plan`, `plan_interval`, `stripe_customer_id`, `stripe_subscription_id`, `subscription_status`, `subscription_current_period_end`, `is_founding_member`
- `subscriptions` table: full subscription lifecycle audit trail (`stripe_subscription_id`, `stripe_customer_id`, `plan`, `plan_interval`, `status`, `current_period_start/end`, `cancel_at_period_end`, `canceled_at`, `amount_cents`, `currency`). RLS policy is read-of-own only; all writes come from the `lemonsqueezy-webhook` Edge Function using the service role key.

- `src/hooks/usePlanLimits.ts` — thin wrapper hooks built on top of `usePlan.ts`: `useProjectLimit`, `useSessionMonthLimit`, `useAnalyticsWindow`, `useCanExportCSV`, `useCanAppearOnLeaderboard`. Each calls `usePlan()` internally instead of duplicating the plan fetch (fixed 2026-07-10).
- `src/components/billing/UpgradeModal.tsx` — pricing UI for all three tiers (Monthly $5, Yearly $39 "Best Value", Lifetime $79 "Founding member"). `handleChoose(planType)` now calls the `create-checkout` Edge Function via `supabase.functions.invoke` and redirects (`window.location.href`) to the returned Lemon Squeezy checkout URL. Shows an inline error banner if the call fails, with per-button loading state. The error banner now surfaces the actual message from `create-checkout`'s response body (added 2026-07-11, via a `FunctionsHttpError` → `error.context.json()` helper) instead of a hardcoded "Could not start checkout" string — supabase-js doesn't put the response body on `error.message` by default, so this had to be read explicitly.

Still doesn't show real post-sync confirmation (e.g. polling until `profiles.plan` actually flips), and there's still no Lemon Squeezy customer-portal link (see "What Is Not Built Yet"). Also fixed 2026-07-14: `usePlan()` now exposes `isLoading` from its underlying query, and `BillingPage` shows a spinner instead of the plan card until it resolves — previously the query's `data: plan = 'free'` fallback meant the page always flashed the Free-plan view first, even for Pro/Founding users, until the real plan loaded.

**Edge Functions (`supabase/functions/`, added 2026-07-10):**
- `create-checkout/index.ts` — accepts `{ planType: 'pro_monthly' | 'pro_yearly' | 'lifetime' }`, derives the user id from the caller's Supabase auth JWT (never trusts a client-supplied id), calls the Lemon Squeezy Checkouts API with the matching variant ID, passes the user id as `checkout_data.custom.user_id`, and returns `{ url }`. **As of 2026-07-15, no longer sends `test_mode: true`** — checkouts default to live mode now that the store and variant secrets are live (see "Current Status"); deployed and confirmed working against a real live purchase 2026-07-16. Also sets `product_options.redirect_url` to `${VITE_APP_URL}/billing?checkout=success` (added 2026-07-11) so Lemon Squeezy sends the user back into the app after a successful purchase instead of its own generic receipt page; skipped gracefully (with a warning log) if `VITE_APP_URL` isn't set. When the Lemon Squeezy API call itself fails (non-2xx), the function now parses the JSON:API error body and returns Lemon Squeezy's own `detail` string in the response (added 2026-07-11) instead of a flat "Failed to create checkout" — this is what let `UpgradeModal` surface the real Lifetime-checkout 404 instead of a generic message. Requires a valid Supabase JWT (default `verify_jwt = true`).
- `cancel-subscription/index.ts` — added 2026-07-14. Looks up the caller's own active subscription (RLS-scoped read, no service role needed for that query), calls Lemon Squeezy's `DELETE /v1/subscriptions/{id}` (a soft cancel — Pro access continues until the current billing period ends via `cancel_at_period_end`, actual downgrade happens later via the webhook's `subscription_expired` handler), and returns a friendly confirmation message. Requires a valid Supabase JWT; the user id is derived from it, never from the request body.
- `lemonsqueezy-webhook/index.ts` — verifies the `X-Signature` header (HMAC-SHA256 hex digest of the raw body, `LEMONSQUEEZY_WEBHOOK_SECRET`) before parsing anything. Handles `order_created` (Lifetime purchase → `profiles.plan = 'founding'`, `is_founding_member = true`; **as of 2026-07-15, also inserts a `subscriptions` row for the order** — see "Current Status" — since this event is the only one for a one-time purchase and previously left the billing-history table with nothing to show for it), `subscription_created`, `subscription_updated`, `subscription_cancelled` (upserts `subscriptions` by `stripe_subscription_id`, updates `profiles.plan` / `plan_interval` / `subscription_status`), `subscription_payment_failed` (marks both rows `past_due`; note this event's `data.type` is `subscription-invoices`, so the subscription is identified via `data.attributes.subscription_id`, not `data.id`), `subscription_expired` (**added 2026-07-14**; fires once a cancelled subscription's grace period actually ends; sets the matching `subscriptions` row's `status` to `'expired'` via the same `payload.data.id` matching as the other `subscription_*` events, then resets `profiles.plan` to `'free'` and `plan_interval` to `null`, skipping Founding/Lifetime members via the same `isFoundingMember` guard used elsewhere), and — **added 2026-07-16, deployed 2026-07-16 (version 17)** — `order_refunded` (fires on a payment refund, whether initiated in the Lemon Squeezy dashboard or by Lemon Squeezy itself; resolves the user via a matching `subscriptions` row keyed by `` `lifetime_order_${orderId}` `` — same key `order_created`'s Lifetime insert uses, so this only matches Lifetime purchases — then `custom_data.user_id`, then (**added 2026-07-16, same day as a deploy-gap fix**) a best-effort match on the most recently updated `subscriptions` row for the order's `customer_id`, for Monthly/Yearly refunds where neither of the first two resolve a user; then unconditionally sets `profiles.plan = 'free'`, clears `plan_interval`/`is_founding_member`, and marks a matched Lifetime `subscriptions` row `'refunded'` rather than deleting it — the `customer_id` fallback only identifies the user, it doesn't mark any row `'refunded'`, since it's not necessarily the exact subscription the order belongs to; see "Current Status" for the caveats this simplification carries). Uses the service role key. **Must be deployed with JWT verification disabled** (`supabase/config.toml` now sets `[functions.lemonsqueezy-webhook] verify_jwt = false`) since Lemon Squeezy never sends a Supabase auth token.
- `supabase/config.toml` was added (didn't exist before) — currently only sets the webhook's `verify_jwt = false`.

**Known limitations in the webhook implementation:**
- `subscriptions.current_period_start` is set from Lemon Squeezy's `created_at` (original subscription creation time), not a true rolling period start — Lemon Squeezy webhook payloads don't expose the current period's start date the way the schema's Stripe-shaped column name implies.
- `subscriptions.amount_cents` is derived from a hardcoded price map (monthly=$5, yearly=$39) matched by variant ID for `subscription_*` events, since those webhooks don't include a price/total field. `order_created`'s Lifetime row is the exception — Order webhooks do include a real `total`, which is used directly (hardcoded `7900` only as a fallback if it's ever missing).
- Existing `stripe_customer_id` / `stripe_subscription_id` columns are reused as-is to store Lemon Squeezy customer/subscription IDs (no schema rename performed — out of scope for this task, still tracked in "Next steps").
- **As of 2026-07-14:** on the `profiles` row specifically (not `subscriptions`), `stripe_customer_id`/`stripe_subscription_id` may silently be left at their previous value instead of the latest Lemon Squeezy IDs, if writing the new values would collide with another profile's existing unique value (see the 2026-07-14 bug fix above). This only affects those two display-only columns — `plan`/`plan_interval`/`subscription_status`/`subscription_current_period_end` are unaffected and always reflect the latest event.
- **As of 2026-07-15:** Lifetime purchases' `subscriptions` row uses `` `lifetime_order_${orderId}` `` in `stripe_subscription_id` rather than a real Lemon Squeezy id, since one-time orders don't have one and the column is `NOT NULL`/`UNIQUE` (see "Current Status" for why). One consequence: `cancel-subscription`'s "most recent active subscription for this user" lookup doesn't know to skip this row — if a Founding member's row were ever the one it picked (it isn't reachable from the UI, since the Founding-member view has no cancel button at all), the Lemon Squeezy API call would 404 the same way a real subscription id would if it didn't exist. Not fixed here since it's not reachable in practice and `cancel-subscription` itself was out of scope for this task.
- **As of 2026-07-16:** `order_refunded` treats every refund as a full downgrade to Free, regardless of whether it was a Lifetime purchase, a full subscription payment, or (in theory) a partial refund on an otherwise-still-active subscription — Lemon Squeezy doesn't distinguish these in a way the handler currently checks. Flagged in a code comment as a simplification to revisit if it comes up in practice, not fixed now.
- **As of 2026-07-16 (fixed same day):** `order_refunded` originally could only resolve the user for a Lifetime purchase refund (via the `lifetime_order_` key) or via `custom_data.user_id` — a Monthly/Yearly subscription refund has neither (that `subscriptions` row is keyed by subscription id, not order id, and `order_refunded`'s payload doesn't carry a subscription id at all, only `customer_id`), so it silently resolved no user and did nothing, returning 200 with no error. Confirmed via a real dashboard-simulated test against a Monthly subscription, and via diffing the actually-deployed function source against the local file (the fix had been written the previous session but never deployed). Fixed with a third, best-effort fallback matching the order's `customer_id` against the most recently updated `subscriptions` row for that customer — this only identifies the user for the profile downgrade, it does not mark any row `'refunded'`, since it isn't necessarily the exact subscription the refunded order belongs to. Carries the same small residual risk as the `custom_data` fallback: if it resolves to a user who is a founding member through a separate, unrelated purchase, this would incorrectly strip that membership. Flagged in code, not fixed now.

## Free Plan Limit Enforcement (as of 2026-07-10)

All five limits are now gated in the UI, each showing `UpgradeModal` with the matching `trigger` when the limit is hit:
- **Projects** (`ProjectsPage.tsx`) — blocks creating a 4th project via `useProjectLimit`.
- **Sessions** (`TimerPage.tsx`, `TimerWidget.tsx`) — pauses the timer and blocks saving past 50 sessions/month via `useSessionMonthLimit`.
- **Analytics** (`DailyView.tsx`, `WeeklyView.tsx`, `MonthlyView.tsx`, `YearlyView.tsx`) — all four period views now blur data older than 7 days via `useAnalyticsWindow` (Daily view was missing this until 2026-07-10; it's aligned with the other three now).
- **CSV export** (`ExportPanel.tsx`) — blocks the export action with a locked banner (uses `usePlan().isPro` directly rather than `useCanExportCSV`, functionally equivalent).
- **Leaderboard** (`ProfileSection.tsx`) — the "Public profile" toggle (which controls leaderboard appearance) is gated via `useCanAppearOnLeaderboard`; viewing the leaderboard itself remains ungated by design (any signed-in user can view `is_public` profiles).

## What Is Not Built Yet

Both of this section's previous "biggest known gap" items — a billing history view and in-app cancellation — shipped 2026-07-14 (see `BillingPage.tsx` and `cancel-subscription/index.ts` above). What's left:
- **No Lemon Squeezy customer-portal link.** The portal URL is available per-subscription in the webhook payload's `urls.customer_portal`, but nothing persists or exposes it — cancellation goes through the in-app flow instead, but there's still no way to update a payment method or view Lemon Squeezy's own invoice PDFs from inside Depthly.

Everything else not yet built:
- The post-checkout success banner is in place, but there's no polling/confirmation that the plan actually changed by the time the banner disappears.
- No automated test coverage for the webhook signature verification or event handling — verified manually against documented Lemon Squeezy payload shapes only.
- No handling for `subscription_paused`/`unpaused` or `subscription_resumed` — only the 7 event types explicitly in scope (the original 5, plus `subscription_expired` as of 2026-07-14 and `order_refunded` as of 2026-07-16) are handled; unhandled events are acknowledged (200) and ignored.
